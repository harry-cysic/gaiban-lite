# 设计note：attention TP4 分片（M4 与 8 卡形态的共同前提）

- 2026-07-21 · **这是设计分析，不是实验**：输入全部是既有实测
  （E2F 的逐张量字节、E3F 的专家/非权重字节、E2F 的 NCCL 单次代价），
  推导部分**未经实测**，不得进入任何"实测"列。
- 结论先行，两条：
  1. **速度上只需做方案 A（只切 o-path）**：A/B/C 三档的净收益
     −1.95 / −1.91 / −1.96 ms/stage，**相差 3% 以内**——多切的字节几乎被
     多加的集合通信原样吃掉。B/C 的复杂度换不到速度。
     A 一项约 **−7.8 ms/token**，是目前最大的单项 decode 杠杆。
  2. **容量上 A/B 都不够，只有 C 装得下 8 卡**（余 +0.49 GiB）。
     所以 **E3F §4 的"分片后 ÷4、余 +0.50"只在 C 档成立**，且它没有计入
     C 的集合通信代价。是否做 C，取决于 §7.7 五行里有没有哪行必须落在 8 卡。

---

## 1. 结构：哪些能分、哪些不能

读 `Ratio4TorchAttention` 的 decode 前向（`ratio4_attention.py`）得到输出路径：

```
sparse_output [b,1,64,512]
  → reshape [b,1,o_groups=8, 4096]          # 每组 4096 值 = 8 个 head
  → einsum("bsgd,grd->bsgr", ·, wo_a[8,1024,4096]) → [b,1,8,1024]
  → flatten → [b,1,8192]
  → linear(wo_b[4096,8192]) → [b,1,4096]
```

**几何上是干净的**：64 head ÷ 4 rank = 16 head/rank = **恰好 2 个 o_group**
（每组 8 个 head）。所以 rank r 拿 head [16r,16r+16) = o_group [2r,2r+2)，
`wo_a` 按组切、`wo_b` 按输入列切，末尾一次 all-reduce 合并 [b,1,4096]。
这是标准 Megatron TP attention，Flash 的 `o_groups=8` 与 TP4 正好整除。

**但 KV 不随 head 分片**：`StaticRatio4KV` 的 `latent` 是
`[seq, LATENT_DIM=512]`，**没有 head 维**（MLA 式：所有 head 共用同一个 latent，
各自用自己的权重展开）。所以每个 rank 仍需完整 latent。
——E3F §4 把 KV 计在"非权重 3.79 GiB"里且未随分片缩小，这一点是对的。

按能否分片给 ratio-4 层的每张量归类（字节为 E2F 实测）：

| 类别 | 张量（ratio-4 层，MB 为 E2F 实测） | MB | 说明 |
|---|---|---:|---|
| **head 分片**（o-path） | `wq_b` 67.109 + `wo_a` 67.109 + `wo_b` 67.109 | 201.3 | **三种层型都有**；末尾需一次 all-reduce |
| head 分片（indexer） | `index_wq_b` | 16.8 | 64 index head，但 topk 须全 rank 一致 → score all-reduce |
| **column-parallel** | `compressor_wkv/wgate`（FP32）、`wq_a`、`wkv`、`index_compressor_*` | 54.5 | 产出**全 rank 共享的状态**，切了要 gather 回来 |
| **必须复制** | `index_weights_proj`、norm/ape/sink | 0.55 | |

三种层型的实测构成（逐张量已量，不再假设）：

| 层型 | 合计 | o-path | compressor | indexer | 其他 |
|---|---:|---:|---:|---:|---:|
| 纯滑窗 | 213.916 | 201.3 | — | — | 12.6 |
| ratio-128 | 230.957 | 201.3 | **8.389×2**（512 宽） | — | 12.9 |
| ratio-4 | 273.183 | 201.3 | **16.777×2**（1024 宽） | 25.7 | 12.6 |

> ⚠️ 初稿把 ratio-128 的 compressor 按 ratio-4 的宽度估算，偏大 16.8 MB。
> **ratio-128 的 compressor 是 512 宽、ratio-4 是 1024 宽**，已实测更正。

## 2. 三个档位（推导，输入全为实测）

| 方案 | ratio-4 层/rank | 倍数 | 新增集合通信 |
|---|---:|---:|---|
| **A：只切 o-path**（`wq_b`/`wo_a`/`wo_b`） | 122.2 MB | 2.24× | +1 次/层 |
| B：A + indexer head | 109.6 | 2.49× | +2 次/层 |
| C：B + column-parallel | 68.7 | 3.98× | +5 次/层 |

## 3. 代价与收益：**三个档位的延迟收益几乎一样**

> ⚠️ **初稿此处有一处硬错**：写的是"head 分片的 all-reduce 顶替原有的那次，
> 净新增 0"。**attention 路径今天一次集合通信都没有**——DP-attention 是全本地的，
> `ratio4_attention.py` 里没有任何 `dist.*` 调用，E2F 的分区统计也显示
> NCCL 只出现在 MoE 区（attention 区 none）。所以分片是**新增**一次 all-reduce，
> 不是顶替。已按此重算。

按 E2F 就地标定的 806 GB/s 与实测 NCCL 单次 ~10.3 µs（B=1），16 卡 stage（11 层）：

| 方案 | 省带宽 | 加通信 | **净** |
|---|---:|---:|---:|
| A | −2.06 ms | +0.11 | **−1.95 ms/stage** |
| B | −2.14 | +0.23 | −1.91 |
| C | −2.53 | +0.57 | −1.96 |

**三者相差 3% 以内。**多切出来的字节，几乎被多加的集合通信原样吃掉。

**这直接决定了实现顺序**：延迟上 A 已经拿到全部收益，B/C 的额外复杂度
**换不到速度**。以 A 的 −1.95 ms/stage 计，四级流水约 −7.8 ms/token，
是目前单项最大的 decode 杠杆（当前 34 ms 量级）。

容量则相反，只有 C 装得下 8 卡：

| 方案 | 8 卡 stage 权重合计 | 余量（卡 23.52 GiB） |
|---|---:|---:|
| 现状 | 26.84 GiB | −3.32 ❌ |
| A | 23.75 | −0.23 ❌ |
| B | 23.63 | −0.11 ❌ |
| C | 23.03 | **+0.49** ⚠️ |

**即：A/B 拿速度但仍装不下 8 卡；C 是唯一能装下的，而它的速度并不更好。**

## 4. 数值：这**不是**逐位改动

E4F/E5F 两次融合都是逐位的，这次不是：

- o_proj 从"单 rank 上 8 组一次算完"变成"4 个 rank 各算 2 组再 all-reduce"，
  **求和顺序变了**；
- indexer 的 `scores.sum(dim=2)` 同理。

按 TARGET §9.6，改变求和序**不可能逐位，别浪费时间追**。所以放行判据是
§1.3 的两条软门：**分数不降（≥494/512）且 `top2_gap` 不越包络（≤0.9595）**，
而不是逐位对拍。这也意味着 E2F 探针的 `--mode ab` 每步逐位对拍**不适用**，
A/B 需要改成"数值见证 + 独立跑长门"。

## 5. 建议的推进顺序

1. **做方案 A（只切 o-path）**：几何干净、改动面最小、**延迟收益已经全拿**
   （−1.95 ms/stage，与 B/C 相差 3% 以内）；
2. 用它跑长门，确认软门通过（这是本项目第一次需要软门放行形态改动）；
3. **B 和 C 押后，且它们的动机只剩容量**——第 3 节表明二者在速度上换不到
   任何东西。C 用"每层 5 次集合通信 + 大改动"换 0.72 GiB，
   **是否值得完全取决于 §7.7 五行里有没有哪一行真的必须落在 8 卡**；
4. 五行的容量需求明确后，才谈要不要 C。

## 6. 未决

- FP32 compressor 能否降到 BF16/FP8——这会同时改善容量与带宽，且与分片正交，
  但是**语义变更**，要独立的质量门。
- 8 卡 prefill 的 workspace 峰值仍未测（E3F §6.1），它可能比权重更早成为墙。
- 本文全部为推导；**任何数字进入 TARGET"实测"列前必须有实验背书**。
