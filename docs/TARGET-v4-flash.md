## 1. 验收标准

验收分两级，**当前阶段只验第一级**：

- **第一级 = 裸引擎，对内部计划值**。这是现在唯一的验收口径，也是本文其余部分的基准。
- **第二级 = serving 框架端到端，对对外承诺**。对外承诺是**框架实测口径**（已含
  30–40% serving 折扣），框架尚未建立，故现在无法验；但它反过来规定了裸引擎必须
  留出的余量。

### 1.1 第一级：裸引擎 vs 内部计划值（当前验收）

| 指标 | **内部计划（裸引擎目标）** | 当前实测 | 达成率 | 状态 |
|---|---|---|---|---|
| 聚合 decode @8K（16 卡方案 B） | **8–12K** | **8,733** | 73–109% | ⚠️ 够到下沿，未达带中/上沿 |
| prefill（双机） | **40K** | **28,622**（C4F） | **72%** | ❌ |
| 单路 decode | **200–350**（+MTP 300–600） | **29.1**（E5F；+MTP ~40） | **8–15%** | ❌ **形态受限**（E2F：当前形态天花板 76.2，须改形态，见 §7.1） |
| 聚合 decode @8K（8 卡方案 A/单机） | **5–8K**（方案 A 双机 10–16K） | — | — | ❌ **形态未跑** |
| prefill（单机 8 卡） | **20K** | — | — | ❌ **形态未跑** |
| 1M 会话并发 | **~25 路** | — | — | ❌ **完全未测** |

**按内部口径，目前只有 decode @8K 一项摸到计划带下沿，其余全部未达或未测。**

### 1.2 第二级：serving 框架 vs 对外承诺（框架建成后）

| 承诺（框架实测口径） | 反推所需裸引擎（÷0.6–0.7） | 对应内部计划 | 一致性 |
|---|---|---|---|
| 单路 ≥150 | 214–250 | 200–350 | ✅ 计划值覆盖 |
| 双机聚合 ≥6,000 | 8.6–10K | 8–12K | ⚠️ 需落在计划带**中上沿**才安全 |
| 单机聚合 ≥2,000 | 2.9–3.3K | 5–8K | ✅ 余量充足 |
| prefill ≥15,000 | 21.4–25K | 20K（单机）/ 40K（双机） | ✅ 双机口径安全 |
| 单机 128K 上下文 | — | 未列 | ❓ 容量问题，见 §7.7 |
| 单机并发 50 路 | — | 未列 | ❓ 按脚注折算 ≈ 单机聚合 2,000，见 §7.7 |

**结构补充（2026-07 客户方案书）**：标准版这几行**不要求同时满足**，
每行可用各自的配置/拓扑——即每行是一个预定义模式，与 §2 同构。
这改变了 8 卡问题的提法：不是"8 卡行不行"，是**五个独立的容量问题**（§7.7）。

**推论：decode 停在 8,733（计划带下沿）时，serving 折扣后约 5.2–6.1K，正好卡在
对外承诺 6,000 的边缘。** 所以 decode 不能以"进了计划带"为由收工，需向带中上沿推进。

### 1.3 质量验收

模型级长 prompt golden 门（D0L）。**放行判据是两条，缺一不可**：

1. **分数不降**：当前基线 **494/512**；
2. **`top2_gap` 不越包络**：基线最大近平局间隙 **0.9595**。

第 2 条比第 1 条更硬——分数 ±3 落在"换一组 prompt 就可能翻号"的范围内，而**越出
包络是质量性变化**。prefill HC 融合正是因 gap 1.4929 越包络被否，chunked prefill
的 chunk=512/999 同理。**容差在任何情况下不得放宽**。

## 2. 部署模型：预定义模式

**不追求完全动态服务。** 交付形态是若干**预定义模式**，每个模式冻结自己的 B 与
max_seq，捕获自己的 CUDA graph 族。模式内需要的是**槽位回收**（一条序列结束后
把新请求填入该行），不是 continuous batching。切换模式是运维动作（重启/重捕），
不是每请求动作。

因此：`B 是图的编译期常量` 不是限制，是设计前提；多档位不叠加显存。

**空白格优先级（2026-07-21 定，profile 完成后于同日修订）**：

0. ~~M4 profile~~ **已完成（E2F）**。结论：M4 受形态限制而非优化不足，
   且发现 §6.1 的带宽天花板算错 4.4×。见 §7.1。
1. ~~8 卡形态验证~~ **已完成（E3F），但结论范围已收敛**：证伪的是
   **8 卡跑吞吐档位**（M7 的 10–16K 聚合），不是"8 卡装不下模型"。
   标准版五行须在 attention 分片落地后**逐行**重算，见 §7.7。
2. **elementwise 尾巴折叠**——E2F 实测 39.5% of replay、每步固定成本；
   **同时利好 M4 与 M2/M3 小 microbatch**，且不改变形态。**E4F 已取下第一块
   （indexer QAT 链，+2.31% 单路，逐位）**，验证了"减少核数≈减少时间"的机制，
   但换算效率约 73%（融合核自身也要时间）。尾巴的绝大部分仍在：
   attention 区 77%、MoE 区 23%，按区域已定位、按链未拆解。见 §7.4。
3. **attention TP4 分片**——**已升级为标准版五行的共同前提**。理由是双重的：
   (a) 单路 ≥150 必须靠它（E2F：当前形态天花板 76.2 tok/s）；
   (b) 8 卡的**任何**档位都必须靠它——E3F 的 1.13 GiB 余量本身就是"分片 +
   FP8 常驻"之后才有的，当前形态连权重都装不下。
   即：它同时决定 M4、8 卡容量与全部单机口径。**当前最高价值的单项形态改动。**
   须评估：每层多一次 allreduce、KV 按 head 摆放、新图族、质量门。
   落地后立即执行 §7.7 的五行逐行重算。
4. **M5 长上下文**（前置：§7.3 的 golden 上限小任务）。
5. M2/M3 推向计划带中上沿（M2 停在下沿则 serving 折扣后卡对外承诺边缘）。

**排序理由（按可交付性，而非按缺口大小）**：裸引擎经 serving 折扣（÷0.6–0.7）后
对上承诺——prefill 17.2–20.0K vs 15K **有余量**；decode 5.2–6.1K vs 6K **卡边缘**；
单路 16.5–19 vs 150 **差一个数量级**。故 prefill 的 28% 缺口是技术完整性问题，
M4 才是交付失败。且**会改变计划的信息应先于不改变计划的优化**：8 卡若跑不动，
M4 家园、M7（计划 10–16K，高于当前方案 B）与单机口径承诺都要重新安排。

**排期便利**：M4 profile 与 8 卡验证都只占**一台机器 8 卡**，另一台整机空闲；
kernel 开发（写、微基准、kernel 级逐位门）多为单卡，可在另一台并行推进
（GPU 独占规则按卡计，跨机不冲突；但 IB/NCCL 类测量仍须错开）。

**M7 值得注意**：报告 §2.1 的方案 A（2× 独立单机 + 前置路由）计划 decode **10–16K**，
**高于**我们正在跑的方案 B（8–12K），且零 IB 依赖；代价是无 1M ctx、KV 池不合并。
它成立与否取决于 8 卡形态，故与优先级 2 是同一件事。

## 4. 硬规则（已被实测确认，勿再挑战）

1. **TP=4，永不 TP8**：Flash 64 头 ÷8 = 8 头/卡会被 pad 到 16（50% 注意力算力白烧），
   且 86 次 allreduce 被迫跨 socket。TP4 = 16 头恰好。TP 只在 socket 内。
2. **P2P 通路必须双重确认**：驱动 patch **且** `NCCL_P2P_LEVEL=SYS`。GPU0–3 处于
   `NODE` 距离，NCCL 默认 P2P level 不含该距离，会**静默回退 SHM（4.12 GB/s vs
   23.79 GB/s）**。⚠️ **`nvidia-smi topo -p2p` 与 `cudaDeviceCanAccessPeer` 在
   NCCL 实际走 SHM 时仍全报 OK，对传输选择没有诊断力**——只有 `isAllDirectP2p`
   与实测带宽算数。每次性能测量须记录并自检（已内建于 result JSON）。
3. **CUDA graph 强制**：B=1 eager 210 ms/步 vs graph 36.3 ms（5.8×）。decode 是
   launch-bound。
4. **质量门必须用长 prompt + 真实权重**：短 prompt（10–22 token）永远不进入 prefill
   大行数路径，会给出无意义的"无损"；合成权重会掩盖 HC 数值问题（真实 hc_scale
   0.03–0.20 vs 合成 1.0）。构造长 prompt 时 **token 数必须精确命中档位**（1024/2048/…），
   差一个 token 就会绕过被测分支。
5. **reference 实现既是 oracle 也是 kernel 来源**：其 tilelang `sparse_attn` 是 prefill
   最大杠杆（6.49×）。不得因 `fp4_gemm` 慢而整体否定 reference kernel。

## 7. 开放问题（按优先级）

### 7.1 M4 延迟模式（有承诺；profile 已完成，见 E2F）

**profile 已做完（E2F，2026-07-21）。它改变了这一格的性质**：M4 不是
"优化没做"，是**形态不支持目标值**。36.3 ms/token 的收支已完全闭合：

| 项 | ms/token | 性质 |
|---|---:|---|
| dense 投影 | 14.02 | 带宽绑定，核已达可达带宽 87%——**只能减字节，不能调核** |
| elementwise 尾巴（2,584 核/replay） | 12.09 | **每步固定成本**（B=1→8 核数不变、时间 +17%） |
| 其余图内（Marlin/NCCL/topk/tilelang） | 4.50 | 各 ≤5% |
| head（eager） | 2.59 | 未归因 |
| 16-rank 上下文固定偏移 | 2.28 | **未归因（E2F-1）** |
| embed + 交接 + token 回环 | ~0.8 | |

**两个互相独立的上限**（推导，输入全为实测）：

| 形态 | 带宽下界 | elementwise 下界 | 上限 |
|---|---:|---:|---:|
| 当前（DP-attention + BF16 常驻 attention 权重） | 13.13 ms | 12.1 ms | **~76 tok/s** |
| 只做 attention TP4 分片 | 4.42 | 12.1 | ~82（被尾巴卡住） |
| 只折叠 elementwise 尾巴 | 13.13 | →0 | ~76（被带宽卡住） |
| **两者都做** | 4.42 | →0 | **~226 tok/s** |
| 再加 attention 权重 FP8 常驻 | 2.97 | →0 | ~337 |

**推论：200–350 的计划值本身没错，错的是"当前形态优化到位即可达成"这一
假设。M4 必须同时做两处形态改动，缺一不可。**下一步不再是 profile，而是：

1. **attention TP4 分片的可行性与代价**（每层多一次 allreduce、KV 摆放、
   新图族与质量门）——这是 latency 模式**独有**的形态，不动吞吐路径。
2. **elementwise 尾巴的成分拆解与融合**——2,584 核、71 种核型、无单一主犯，
   需先按相位定位。**对 M2/M3 吞吐同样有价值**（它是"小 microbatch 交织
   不划算"的机制，现已量化）。

- 延迟模式有**自己的图族、kernel 几何与拓扑**，可独立于吞吐路径特化。
- 家**可能**在 8 卡 TP4×PP2（报告：单用户是单机游戏，加机器不涨 batch-1），
  但**当前形态装不下**（E3F），而 M4 恰恰是 KV 需求最小的档位，
  分片后是否装得下**未测**——属 §7.7 的第 2 行。现有全部实测是 16 卡口径。
  ⚠️ 8 卡形态下 stage 是 21–22 层而非 11 层，每 stage 字节数近似翻倍；
  重问时须带 E2F/E3F 的字节账重算，不能沿用 16 卡的 per-stage 数。

### 7.2 8 卡 TP4×PP2 — **当前形态已验，不成立；分片后须逐行重问（见 §7.7）**

实测（titan064，B=1，max_seq 3328，即最省配置）：22 层 stage **加载到第 19 层
OOM**；每卡实测最多 18 层，PP2 需要 22 层。**在当前驻留形态下**权重本身
23.05 GiB > 卡容量 23.52 GiB 减去必需的非权重 3.79 GiB。

**⚠️ 这个结论的范围**：证伪的是"**8 卡跑吞吐档位**"，不是"8 卡装不下模型"。
E3F 自己的字节账就给出反例：attention TP4 分片后 23.02 GiB（余 0.50），
再加 FP8 常驻 22.39（余 1.13）——**模型装得下**，装不下的是吞吐档位所需的
大 KV 池。此前本节写成"三条线全灭"是**超出证据边界**的表述，已收敛。

因此：
- **M7 方案 A（10–16K 聚合）已证伪**（移入 §5.2）——该档位的 KV 池远超 1.13 GiB；
- **8 卡的其余档位（长上下文、单路延迟、prefill）是未被正确提问的问题**，
  不是已关闭的问题。见 §7.7；
- routed experts 的 17.53 GiB 已是 FP4 且已分片，**无压缩余地**——所以
  8 卡的一切可能性都取决于 attention 侧形态。

未了结：prefill 侧 workspace 峰值未测；TP8×PP1 未探索（§4.1 否决 TP8 的
前提是 attention 被 TP 分片，在 DP-attention 下该推导不直接适用）。

### 7.3 M5 长上下文完全未测
128K/512K/1M 的容量、吞吐、以及"25 路 1M 会话并发"均无数据。这是双机方案的
核心卖点之一。

**前置障碍：reference 侧 golden 上限只有 4096 token**（见 §8），且**与权重放在
哪台机器无关**——根因是 `model.py:685` 的 `hc_post` 广播 `[b,s,hc,hc,d]` fp32
= 256 KiB/token，8192 需要恰好 2.00 GiB 的**单次分配**。没有更长的 golden，
长 ctx 的正确性就没有模型级背书。

**已识别的解法（未做）**：`hc_post` 在 token 维上逐行独立，**沿 s 分块计算是
数学精确的**（不是近似）。改动只在 reference 侧评测 harness，不触碰 runtime 语义，
也不影响任何已冻结的 golden。叠加已发现的 `n_mtp_layers` 冗余（省 0.52 GiB，
见 §8）后，长 prompt golden 的天花板应能显著抬高。**这是 M5 正确性验证的前置
小任务，建议先于 M5 性能实验完成**——否则 M5 只能产出无质量背书的性能数字。

生成新 golden 需要 reference MP=8 权重（`~/Workspace/DeepSeek-V4-Flash-mp8/`，
两台 titan 均已具备）；跑已冻结的门则不需要，golden 是 JSON。

### 7.4 已知可回收的性能
- **vLLM fused HC 在 ≥1024 行数值错误**（rel_fro 2.3e-4 → 1.07e-1，公有 API 无法
  绕开；已用 MAX_ROWS=896 规避但精度仍不足）。**正确实现（自研或上游修复）可
  取回 prefill +19.9%（→30,345）**。分派点：`vllm/model_executor/kernels/mhc/
  tilelang.py:43`；通用路径 `hc_prenorm_gemm_tilelang` 已验证正确。
- MoE 集合重叠：**已结案**，天花板仅 51.8% 重叠率（NCCL P2P 在 NODE 距离 PCIe 上
  是 SM 驱动，与 Marlin 抢 SM），零开销实现也只到 +4%，不值得继续投入。
- **tilelang 稀疏核的包装层搬运税 16.1%**（C4F）：sm89 逼出的 `head_chunk=16` 循环
  每块要 `.contiguous()` + `copy_` 回写；但 64 头 / o_groups=8 意味着一个 chunk
  恰好是 2 个 o_group，**出口侧可按 head chunk 分块而不必拼回**，该税可整块去掉。
  核本体另有余量：**31% MFU**（51.6 TFLOPS）。
- **prefill 当前归因**（C4F，分母 1.151 s）：MoE 37.5%、**HC 27.2%**、ratio-4 18.6%、
  ratio-128 13.8%、norm 2.8%。HC 升至第二桶，使上面那条 vLLM kernel 缺陷的修复
  价值高于此前评估。

- **decode elementwise 尾巴 39.5%**（E2F）：一次 stage replay 有 2,584 个未融合的
  eager 逐元素核（71 种核型、平均 1.18 µs、无单一主犯），共 3.06 ms/replay，
  **每步固定**（B=1→B=8 核数 1,980→1,983、时间仅 +17%）。四级流水下 12.1 ms/token。
  这是 decode 侧当前最大的单一可攻项，且**对 M4 与小 microbatch 吞吐同时有效**。
  **尾巴按核数收费不按字节收费**（平均 1.03–1.73 µs，都在 4090 最小 kernel 时长
  附近），所以减少核数的融合大体换成时间（E4F 实测换算效率约 73–89%，非 1:1）。
  区域定位：**77% 在 attention 体内**（213.8 µs/层、180.6 核/层），23% 在 MoE 区。
  **按链的待攻排序见 E2F §5b**（ratio-4 层非 GEMV 234 µs/层、ratio-128 122 µs/层，
  一个 stage 约 1.65 ms，E4F 已取 0.29 ms）；下两个目标是 `index_topk_done`
  （44.5 µs/层）与 `raw_kv_done`（30.1）。该表是**排序用估算**，引用前须层内 A/B。
- ~~KV-latent FP8 QAT 链~~ → **已兑现并设为默认（E5F）**：`fp8_quant_dequant`
  在 decode 是 ~15 µs/层的 launch-floor 工作，且**每种层型都跑**（不像 indexer
  链只在 ratio-4 上）。融合核逐位（自检 18/18、层内 480 步、D0L 长门与冻结基线
  逐位一致），层内成对交替 A/B **−2.25%/stage**，闭环 **28.402 → 29.055 tok/s
  （+2.30%）**。prefill 无可测影响（1.9897→1.9922 s，ratio-128 分量墙不变）。
  **累计 E4F+E5F：27.740 → 29.055，+4.74%，两步都逐位。**
  ⚠️ 同一段代码在 prefill 只占 0.14%（C4F 冻结分相）——**launch-floor 绑定是
  decode 现象**，不要按 prefill 的占比给 decode 的链排序，反之亦然。
- ~~indexer QAT 链~~ → **已兑现并设为默认（E4F）**。E2F 测得该链在 B=1 值
  replay 的 4.3%（**§2 此前先验估算"0.2–0.4%"低了一个数量级**——这正是当初要求
  "用 profile 定夺而不是直接启用"的理由）。E4F 把 C4F 的融合核接到 decode 路径：
  **逐位相等**（kernel 自检 / decode 形状 max_abs_diff 0.0 / 层内 480 步 /
  D0L 长门复现冻结的 494/512 且 max `top2_gap` 0.959503173828125 与八条 prompt 的
  首个失配 logit 全部逐位一致），层内成对交替 A/B **−3.82%/stage**，
  全模型闭环 **27.740 → 28.381 tok/s（+2.31%）**。已设为默认，
  `DSV4_INDEXER_QAT_DECODE=ref` 可回退。
  ⚠️ 注意收益机制与 prefill 完全不同：不是省带宽，是省 kernel 启动，
  **C4F 的 90.5× 不可折算**。
- **shared expert 走 BF16 反量化副本**（E2F）：`moe_runtime.py` 的
  `shared_path = "bf16_dequant_correctness_fallback"`，FP8 常驻副本同时存在但热路径
  不读。改用 FP8 可省 6.291 MB/层 = 270 MB/token（总字节的 2.2%），需质量门。
- **attention 权重在 runtime 是 BF16、compressor 是 FP32**（E2F），而 checkpoint
  非专家权重是 FP8。§3 的"无 BF16 税"说的是 on-disk，**runtime 侧有**。FP8 常驻
  可把 attention 字节减半，需质量门。

**自研优先于等上游**：C4F 已证明本项目能写出 90.5×、kernel 级+层级双逐位过门的
融合 Triton 核。HC 边界核走自研（保持归约序即可逐位）比 vendor 一份 vLLM 补丁或
等上游修复更干净——vLLM 那条路已被长门否掉（规避方案精度不足）。

### 7.7 标准版五行：8 卡容量须**逐行**重算（attention 分片落地后）

**外部口径（仓库外，2026-07 客户调研版方案书）**：标准版"单机 · 8 卡 RTX"
一列承诺五项。**关键结构：这五行不要求同时满足**，允许各自用不同的配置/拓扑
达成——即每行对应一个预定义模式，与 §2 模式矩阵同构。方案书脚注亦明写
"生成速度与并发能力随上下文长度、负载模式有所变化""并发请以聚合吞吐折算"。

| # | 承诺（框架实测口径） | 该行的容量自由度 |
|---|---|---|
| 1 | 上下文 **128K** | B 可以很小；KV 池按单路/少路预算 |
| 2 | 单路 **≥150** tok/s（典型 150–250） | B=1；KV 最小；**受 §7.1 的两个形态改动共同支配** |
| 3 | 聚合 **≥2,000** tok/s | 大 B；KV 池最大——**§5.2 已证伪的就是这一行** |
| 4 | 并发 **50 路以上** | 按脚注以聚合吞吐折算；50 路 × 40 tok/s ≈ 2,000，与第 3 行同一工作点 |
| 5 | 文档解析 **≥15,000** tok/s | prefill；KV 需求低但 workspace 峰值高（未测） |

**要做的事**：attention TP4 分片落地后，对这五行**逐行**重算 8 卡容量，
每行给出**冻结配置（B / max_seq / KV dtype / 分片程度）+ 可行性 + 余量**，
而不是一个笼统的是否。E3F 已提供逐层驻留字节（§3 常数表），是这五次计算的输入。

**为什么必须逐行**：第 3 行（大 KV 池）被证伪不蕴含第 1/2/5 行被证伪——
它们的 KV 需求相差一个数量级以上。把五行合并成一个"8 卡行不行"，正是
E3F 初稿犯的错。

⚠️ 外部另有粗估认为"每行都紧、但都不是不可能"。**该估算未经实测，
仅用于判断是否值得重问，不得写入任何'实测'列，也不得作为规划依据。**

## 9. 测量方法学（硬约束——决定本文数字的真假）

这些不是建议。违反其中任何一条得到的数字**不得写入本文的"实测"列**。

1. **4090 上串行 A/B 计时不可信**：同配置三轮可给出 51/219/318 µs 三个完全不同的
   差值（时钟/热漂移）。所有 graph-replay 级 A/B 必须用**成对交替重放**（两条 lane
   常驻、每步背靠背、逐步交替顺序）。MTP 与 HC 的可信数字全部出自此口径。
2. **`synchronize` 式分组件计时的开销随 forward 次数增长**（1024 段长的臂上 +14.7%），
   故**不能跨 forward 次数不同的臂相减**；headline 一律取未插桩 p50。正例是 C4F 的
   分相探针：用 CUDA event 而非 host sync，插桩开销 0.13%、相位覆盖 100.02%。
3. **一个组件的桶会吸收整条链的跨 rank 偏斜**。C2F 的 MoE 桶曾被记成 131 ms/层，
   隔离测只有 49.8——因为 MoE 是链上唯一含集合通信处。**某桶异常大时，先问"它是不是
   链上唯一同步点"，再怀疑该组件本身。**
4. **微基准 ≠ 层内，只用于判活/判死，不用于折算收益**。C4F 的 cast 提取微基准预期
   0.65 ms/层、层内只兑现 0.166；A5F 的 HC 微基准 2.92× 在集成态只回收 4.7 ms/stage。
5. **≥3 轮并报轮间离散**。已放行数字的离散在 0.03–0.29%；**超过 1% 说明口径里有未控
   变量**（经验上是 NCCL 传输选择、分配器状态、或插桩本身）。
6. **改变行集合/行序的改动不可能逐位，别浪费时间追**：Marlin 分组 GEMM 对 batch
   组成是 1–2 ULP 敏感的（行分组变则求和序变），这是固有性质不是 bug。失步投机、
   集合重叠的行分块、chunked prefill 均属此类——它们只能走"不越 `top2_gap` 包络"的门。
7. **跨 TP lane 输出本就不逐位**（reduce_scatter 求和序），规范输出取 tp_rank0。
   **用种子残留态做"lane 一致性"检查会得到假失败**（真实 prompt 下一致）。
8. **HC 类数值门只在真实权重上有意义**：真实 `hc_scale` 0.03–0.20 会压制 GEMM 误差
   向 post/comb 传播，合成 `hc_scale=1` 放大约 4×（A5F 的"≤1e-5"即由此而来的假象）。
9. **`e0ff` 的种子 20260717 是筛出来的**（全部抽样落在 E2M1 格点上），exact-topk 门
   对格点翻转敏感，**不要随手改种子**。
10. **对照实验的每个臂必须有独立产物路径**。`run_*.sh` 的输出名多由配置参数
    决定，两个只在 env 上不同的臂会写同一个文件、互相覆盖，最后两份"结果"
    字节相同——**与"处理无效"同形**。E5F 因此差点误判，且顺手覆盖了 C2F 的
    冻结产物（已还原）。跑对照前给每臂独立 out-dir，并先确认不会写到
    `runtime/out-*/` 下被跟踪的冻结产物上。
11. **处理组必须有见证**。env 门控的变体经 ssh 传到远端时会静默失效
    （单引号 `ENV_BASE` 里的 `${VAR:-:}` 不在本地展开，到远端变成空操作），
    **症状与"该杠杆无效"完全同形**。E4F 因此差点把一个实测 −3.82% 的融合
    读成 0%。凡是 A/B，结果 JSON 必须记录**实际解析出的处理组配置**
    （`e1f_full_decode_bench.py` 的 `attention_modes` 即此用途），
    否则"没生效"会被当作"没效果"写进结论。
12. **等待/监视类循环必须能自己失败**。这类循环写错时**不报错，只静默地永远等下去**，
    表现得像"还在跑"——比崩溃危险，已在本项目咬过两次（goal 会话卡在等 18 个只会
    产生 16 个的结果文件；mp8 复制的监视器因 `pgrep` 自匹配在传输早已完成后继续空转）。
    三条要求：
    - **有超时上限**，超时即报告而非继续等；
    - **有"条件已达成但循环未退出"的自检**（例如同时检查目标产物是否已就绪，
      而不只检查进程是否存在）；
    - **`pgrep` 模式串必须规避自匹配**（用 `partia[l]` 式括号写法），否则会匹配到
      承载该循环的 shell 自身。参见 `HANDOFF.md` §3.3.2。
    等待条件应尽量绑定**产物**（文件出现/字节数达标）而非**进程存在性**——进程视角
    既会自匹配，也无法区分"正常结束"与"异常退出"。
