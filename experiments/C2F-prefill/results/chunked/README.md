# 分段 prefill 在 C2F 吞吐口径下的实测:C3F 的 1.5–2.3× **不兑现**

第二十六竖条(2026-07-21,titan064 实测)。C3F([`C3F-chunked-prefill`](../../../C3F-chunked-prefill/README.md))
在 E2E 长门语境下测得"分段比整段快 1.5–2.3×、stage-0 峰值 20.671→15.091 GiB",
并明确标注**那不是 C2F 吞吐口径**。本竖条把真分段接进 C2F bench,在同口径下判定。

## 结论速览

| 项 | 结果 |
|---|---|
| **整段控制臂** | **25,307** input tok/s(3 轮 25340/25301/25281,离散 0.23%)—— 与冻结值 **25,308** 差 **0.004%**,口径与环境完全对齐 |
| **分段吞吐** | **全部慢于整段**,且随段数单调变差:8192 切 2 段 **0.990×**、4 段 **0.970×**、8 段 **0.958×**。**最好的分段方案也没有正收益** |
| **C3F 的 1.5–2.3× 在本口径下** | **不兑现,方向相反**。最差一格 −4.2%(8×1024) |
| **"长整段有超线性尺寸罚金"** | **证伪**。整段臂每 token 成本随 L **单调下降**(1024→8192:161.2→158.1 µs/token/lane,−2.0%),是**规模经济**而非罚金 |
| **规律** | 每 token 成本几乎只由**单次前向的行数 S** 决定,与总长 L 和已累积上下文基本无关(同 S across L 离散 ≤1.31%) |
| **罚金拆分** | 分段慢的两项可加(L=8192):(a) 丢掉大前向的规模经济 +0.7%~+2.5%;(b) chunk 分支自身开销 + 上下文税 +0.3%~+2.3%(到 16K/16 段为止 ≤+3.3%) |
| **显存** | **真实且可观**:8192 峰值 **20.409 → 14.095 GiB**(−6.314,−30.9%),收尾 free **2.277 → 8.650 GiB** |
| **显存拆分** | 激活 **4.125 GiB(65.3%)** + MoE per-shape 缓冲 **2.188 GiB(34.7%)**,两项相加 6.313 = 实测 6.314 |
| **容量墙(矩阵外补测)** | **16K 整段 4 rank 全 OOM**;16×1024 分段峰值仅 **14.382 GiB**、还剩 8.375 GiB。整段天花板反解 **≈11.4K token** |
| **与 C3F 的差异根因** | C3F 的整段臂注册了**4 个** MoE 形状(3 种 prompt 长度 + decode)且贴着 OOM 跑;C2F bench 两臂各只注册**1 个**形状。C3F 的"缓冲占一半"含**形状数塌缩**,本竖条只有**尺寸效应** |
| **prefill 头条** | **不更新,保持 25,308**(本竖条复现 25,307)。单池 T 保持 **2,322** |
| **同池共存** | 判断**改变**:整段 prefill 需在 decode 之上再要 **7.54 GiB/卡**(decode 前沿只剩 **0.531 GiB**),结构上不可能;分段到 1024 后降到 **1.229 GiB**,可用 **~10 个 bl 单位**的 decode 退让换来(≈ −8% decode 吞吐)。**从"不可能"变成"要付明确代价"** |

**一句话**:分段 prefill 在这台机器上是**显存/容量杠杆,不是吞吐杠杆**。
它稳定地要付 1–4% 吞吐,换回 30% 峰值显存,并且把 prefill 峰值从
**O(总长)** 变成 **O(段长)** —— 后者才是它不可替代的地方:16K 整段直接 OOM,
16×1024 分段只用 14.4 GiB。C3F 观察到的加速**没有**在同口径下复现,
其成因已定位到 C3F 整段臂自身的形状注册与容量压力(§5),不是分段的内在收益。

## 0. 口径与实现

### 0.1 bench 的 `--prefill-chunk`

C2F bench 的 `--chunk` 历史上 = **一次整段 `start_pos=0` prefill 的长度**,
从来不是"把 prompt 切成几段喂"。本竖条给
[`runtime/c2f_prefill_stage_bench.py`](../../../../runtime/c2f_prefill_stage_bench.py)
加了 `--prefill-chunk`:

- `--chunk L` 现在是**总 prefill 长度**;
- `--prefill-chunk 0`(默认)= 一次整段前向 —— **与冻结路径逐行相同**,
  `global_row_shapes` 仍是 `(4L,)`,所以既有产物全部原样复现;
- `--prefill-chunk S`(>0)= 切成连续的 `start_pos>0` 多 token 前向,
  复用 C3F 的增量能力(`dsv4_direct/chunked_prefill.py` + 三个 `__call__` 的
  `is_chunk` 分支);
- **MoE per-shape 缓冲按段长注册**(`{4S}`),与 C3F 的
  `e0ef2e_golden_gate.py` 同一处理;
- 吞吐口径**不变**:`input tok/s = 4L / 整个 prefill 的墙钟`,两种模式因此
  在**等交付 token 数**下直接可比。

### 0.2 分段确实发生了 —— 证据

每份结果 JSON 记录 `prefill_forwards` / `segment_lengths` /
`segment_start_positions` / `global_row_shapes` /
`moe_registered_global_rows`(纯观测,像 C3F 那样):

| L | `--prefill-chunk` | `prefill_forwards` | `segment_lengths` | 注册的 MoE 全局行 |
|---:|---:|---:|---|---|
| 8192 | 0 | 1 | `[8192]` | `[32768]` |
| 8192 | 4096 | **2** | `[4096, 4096]` | `[16384]` |
| 8192 | 2048 | **4** | `[2048]×4` | `[8192]` |
| 8192 | 1024 | **8** | `[1024]×8` | `[4096]` |
| 4096 | 1024 | **4** | `[1024]×4` | `[4096]` |
| 2048 | 1024 | **2** | `[1024, 1024]` | `[4096]` |

外加 `segment_walls_s`(每段墙钟)——见 §2.3,它直接把"上下文税"量出来了。

### 0.3 冻结配置(与产出 25,308 的那次逐项相同)

11 层 L11–L21、`--moe-mode w4a8`、`--indexer fused`、
`--sparse-backend tilelang`、`--row-block 1024`、`--sparse-head-chunk 16`、
`iters 5 / warmup 2`、`seed 20260721`、`input_scale 0.02`、
`PYTORCH_CUDA_ALLOC_CONF=expandable_segments:True`、
**`NCCL_P2P_LEVEL=SYS`**、torch 2.11.0+cu130、单机 titan064 GPU0–3。

> **一处必须如实记下的口径偏差**:任务描述里写的是"tilelang attention +
> **fp8 KV** 的已放行配置"。C2F bench **没有 KV dtype 旋钮**——它不给
> `build_physical_stage` 传 `kv_dtype`,取默认 **bf16**。冻结的 25,308 也是
> bf16 KV。本竖条为保持同口径**照搬 bf16**,没有引入 fp8 KV 这个新变量。
> 若要 fp8 KV 下的分段结论,需先给 bench 加该旋钮并重取整段基线。

### 0.4 P2P 自检(每份 JSON 自带)

90 个探针样本(45 次总形状 + 45 次段形状):
**min 8.18 / p50 23.20 / max 23.94 GB/s,落在 SHM 区(<6 GB/s)的 0 个**。
唯一一个 8.18 是 `r1/c2f-chunk4096-...-seg2048.json` 的一次 reduce_scatter 采样
(同 run 的 all_gather 22.61,同点另外两轮 22.9–23.1),属探针抖动(p50 只取 3 个
样本)而非传输回退——SHM 回退是**系统性**的 ~4 GB/s,不会只影响一次采样。

## 2. 规律:每 token 成本只由**单次前向的行数**决定

把每点换算成 **µs / token / lane**(= `wall p50 / L`),规律非常干净。

### 2.1 整段臂:**没有超线性罚金,是规模经济**

| 整段长度 L | µs/token/lane | tok/s |
|---:|---:|---:|
| 1024 | 161.238 | 24,808 |
| 2048 | 161.945 | 24,700 |
| 4096 | 159.142 | 25,135 |
| **8192** | **158.057** | **25,307** |

L 涨 8×,每 token 成本**降** 2.0%。这与 C2F 主 README 早已记录的
"prefill 吞吐对 chunk 基本平坦(O(s²) 项在 8K 内尚不主导)"一致,
并把它推进了一步:**不只是平坦,是轻微下行**。

> 所以 C3F 提示的"长整段有超线性尺寸罚金"在 C2F 口径下**证伪**。
> 需要量化的不是罚金曲线,而是**规模经济曲线**:上表就是它。

### 2.2 分段臂:每 token 成本 ≈ **同长度整段**的成本

按**单次前向行数 S** 分组(每组含不同总长 L):

| 前向行数 S | 均值 µs/token | 组内离散 | 逐点(L=2048 / 4096 / 8192) |
|---:|---:|---:|---|
| 1024 | 164.011 | **1.31%** | 162.985 / 164.106 / 164.942 |
| 2048 | 162.449 | **0.75%** | — / 162.480 / 162.923 |
| 4096 | 159.416 | **0.48%** | — / — / 159.690 |
| 8192 | 158.057 | 0.23% | — / — / 158.057 |

**同一个 S、不同总长 L,成本几乎相同(≤1.31%)。** 也就是说:
把长度 L 的 prefill 切成 S 长的段之后,它的每 token 成本就**塌到 S 长整段的
成本水位**上——而那个水位比 L 长整段的水位**高**(§2.1 的规模经济反过来咬)。
这就是分段必然亏的原因,与"段数"是同一件事的两种说法。

### 2.3 上下文税:实测 +0.3%~+2.3%,即 O(s²) 项的真实大小

`segment_walls_s` 记录每段墙钟。L=8192 切 8×1024(instrumented pass):

```
[188.4, 196.5, 193.2, 193.1, 193.4, 194.0, 193.6, 194.1] ms
```

第 8 段(要 attend 8192 的上下文)与第 1 段(只有 1024)**几乎等价**。
把它与**独立的整段 1024**(161.238 µs/token)对照,就把上下文税单独量出来了:

| 段长 S | 段数 | 分段臂 µs/token | 独立整段同长 | **上下文税** |
|---:|---:|---:|---:|---:|
| 1024 | 2 (L=2048) | 162.985 | 161.238 | **+1.08%** |
| 1024 | 4 (L=4096) | 164.106 | 161.238 | **+1.78%** |
| 1024 | 8 (L=8192) | 164.942 | 161.238 | **+2.30%** |
| 1024 | **16 (L=16384)** | 166.612 | 161.238 | **+3.33%** |
| 2048 | 2 (L=4096) | 162.480 | 161.945 | **+0.33%** |
| 2048 | 4 (L=8192) | 162.923 | 161.945 | **+0.60%** |
| 2048 | **8 (L=16384)** | 164.557 | 161.945 | **+1.61%** |
| 4096 | 2 (L=8192) | 159.690 | 159.142 | **+0.34%** |

段数翻倍,上下文税约 +0.6~+1.0 个百分点。**这是 O(s²) 项 + chunk 分支自身
开销(并集布局、ring 快照、绝对位置掩码)的合计**,到 16K / 16 段为止
合计不超过 **3.3%**,且**没有出现拐点**——O(s²) 项在本工作点仍不主导。

## 3. 罚金拆分:两项可加

以 L=8192 为例,把"分段比整段慢"拆成两项(全部用无插桩 p50):

| 分段方案 | 实测 wall | (a) N×同长整段 | **(a) 规模经济损失** | **(b) chunk 分支 + 上下文税** | 合计 |
|---|---:|---:|---:|---:|---:|
| 2×4096 | 1.3082 | 2 × 0.6518 = 1.3037 | **+0.69%** | **+0.34%** | **+1.03%** |
| 4×2048 | 1.3347 | 4 × 0.3317 = 1.3267 | **+2.46%** | **+0.60%** | **+3.08%** |
| 8×1024 | 1.3512 | 8 × 0.1651 = 1.3209 | **+2.01%** | **+2.30%** | **+4.36%** |

(基准 = 整段 8192 的 1.2948 s;(a) = N×同长整段 ÷ 整段 8192,
(b) = 实测 ÷ N×同长整段。)

两项的**含义**:

- **(a) 规模经济损失**:纯粹因为"一次前向的行数变小了"。它与分不分段无关——
  独立跑一个 2048 整段 prefill 也是这个成本。**这是 §2.1 曲线的直接后果。**
- **(b) chunk 分支 + 上下文税**:增量路径自己的开销。**这是分段独有的部分**,
  也是本竖条唯一新增的成本。它在 8K 内 ≤2.3%。

> 4×2048 的 (a) 比 8×1024 的 (a) 还大(+2.47% vs +2.01%),因为整段 2048 的
> 每 token 成本(161.945)反而略高于整段 1024(161.238)——§2.1 曲线在低端
> 有一处非单调。如实记录,不修饰。

|---:|---:|---:|---:|---:|---:|---:|---:|---:|

| 8192 | 1024 | 0.3506 | 0.2270 | 0.4827 | 0.4564 | 0.0223 | 1.5464 | **+14.7%** |
| 4096 | 整段 | 0.1757 | 0.0838 | 0.2207 | 0.1620 | 0.0146 | 0.6578 | +1.0% |
| 4096 | 2048 | 0.1759 | 0.0950 | 0.2263 | 0.1787 | 0.0070 | 0.6848 | +2.9% |

| 2048 | 整段 | 0.0872 | 0.0472 | 0.1171 | 0.0878 | 0.0037 | 0.3438 | +3.5% |

⚠️ **插桩开销随前向数暴涨**(`timed()` 每个分量前后各一次
`torch.cuda.synchronize`;段越短,CPU 跑在 GPU 前面越多,每次 sync 越贵)。
1024 段长的臂被插桩多算了 **~14.6%**,所以**这张表只能读趋势,不能读绝对值**;
headline 一律取无插桩 p50。

趋势本身在**插桩失真最小**的一格(8192 整段 +0.3% vs 2×4096 +0.7%)最可信,
增量 +0.0198 s 的构成:

| 分量 | Δ(2×4096 − 整段) | 方向 |
|---|---:|---|
| attention ratio128 | **+0.0102** | 变差(chunk 分支的并集布局) |
| moe | **+0.0093** | 变差(GEMM 与集合都变小) |
| hc | +0.0077 | 变差 |
| attention ratio4 | **−0.0054** | 略变好 |
| norm | −0.0030 | 略变好 |

**MoE 变差是可预期的**:分成 N 段就是把 1 次 `[4L,4096]` 集合换成 N 次
`[4L/N,4096]`,而小集合的总线效率更低——本竖条自带的段形状自检直接量到了
(段形状 4096 行时 19.7–22.7 GB/s,总形状 32768 行时 23.4–23.8 GB/s)。

## 4. 显存(真实收益所在)

两个口径都从拉回本地的 JSON 复核。`prefill peak` = 整个计时循环的 allocated
峰值;`activation` = `prefill peak − after load`,即**前向自己**在常驻权重之上
要的量。

| L | 分段 | MoE 行 | **MoE 缓冲** | after load | **prefill peak** | **activation** | 收尾 free |
|---:|---:|---:|---:|---:|---:|---:|---:|
| **8192** | **整段** | 32768 | **2.501** | 15.368 | **20.409** | **5.041** | **2.277** |
| 8192 | 4096 | 16384 | 1.250 | 14.117 | 16.909 | 2.791 | 5.818 |
| 8192 | 2048 | 8192 | 0.625 | 13.492 | 15.033 | 1.541 | 7.719 |
| **8192** | **1024** | 4096 | **0.313** | 13.180 | **14.095** | **0.916** | **8.650** |
| 4096 | 整段 | 16384 | 1.250 | 14.117 | 16.640 | 2.523 | 6.078 |
| 4096 | 2048 | 8192 | 0.625 | 13.492 | 14.890 | 1.398 | 7.859 |
| 4096 | 1024 | 4096 | 0.313 | 13.180 | 13.952 | 0.773 | 8.793 |
| 2048 | 整段 | 8192 | 0.625 | 13.492 | 14.756 | 1.264 | 7.986 |
| 2048 | 1024 | 4096 | 0.313 | 13.180 | 13.881 | 0.701 | 8.879 |
| 1024 | 整段 | 4096 | 0.313 | 13.180 | 13.814 | 0.634 | 8.943 |
| *16384* | *整段* | *65536* | — | — | **OOM** | — | — |
| *16384* | *2048* | 8192 | 0.625 | 13.492 | 15.319 | 1.827 | 7.422 |
| *16384* | *1024* | 4096 | 0.313 | 13.180 | **14.382** | 1.202 | **8.375** |

(单位 GiB,单 rank;四个 rank 逐字节相同。斜体行为矩阵外补测,见 §4.3。)

注意 `after load` 只有四个值(13.180 / 13.492 / 14.117 / 15.368),完全由
**段长**决定,与总长 L 无关——L=4096 和 L=16384 切成 1024 段后 `after load`
都是 **13.180**。这直接证明常驻侧的差异**全部**来自 MoE per-shape 缓冲
(§4.1),KV 状态随 `max_seq_len` 的增长可忽略。

### 4.1 拆分:激活 65% / 缓冲 35%,两项相加闭合

L=8192,整段 → 8×1024:

| 项 | 整段 | 8×1024 | Δ | 占比 |
|---|---:|---:|---:|---:|
| MoE per-shape 缓冲 | 2.501 | 0.313 | **−2.188** | **34.7%** |
| 前向激活 | 5.041 | 0.916 | **−4.125** | **65.3%** |
| **合计(= peak 差)** | **20.409** | **14.095** | **−6.314** | 100% |

−2.188 + −4.125 = **−6.313**,实测 peak 差 **−6.314** —— **闭合到 1 MiB。**

拆分是**可分离的,而且是直接测的**,不是估的:

- **缓冲项**独立可测:`moe_shape_buffer_bytes` 直接把
  `_MarlinBuffers` 的全部张量按 storage 去重求和。四种形状实测
  0.6252 / 1.2504 / 2.5008 GiB 与 0.3126 GiB,对行数**严格线性**:
  4096 / 8192 / 16384 / 32768 行各自 **80.04 / 80.03 / 80.03 / 80.03 KiB/行**
  (与 C3F 记的 "~80 KiB/行" 精确一致)。
- **它整个落在 `after load` 里**:`after load` 差 15.368 − 13.180 = **2.188 GiB**,
  与缓冲差**完全相同**——即 `after load` 的全部差异就是 MoE 缓冲,权重与 KV 状态
  在两臂间不变(KV 几乎不随 `max_seq_len` 变:ratio-4 的 raw 是 128 槽定长 ring,
  压缩行只有 `max_seq_len/4`)。
- **激活项**因此就是余下的 `prefill peak − after load`,实测对**段长**近似线性:
  0.916 / 1.541 / 2.791 / 5.041 GiB 对 1024 / 2048 / 4096 / 8192 行,
  即 **≈0.576 GiB / 1024 行 + 0.34 GiB 常数**。

### 4.2 与 C3F 的"缓冲占一半"对账 —— 两者**不是同一件事**

C3F 记 4096 提示省 3.70 GiB = 1.876(缓冲)+ 1.828(激活),缓冲占 **50.7%**;
本竖条 8192 省 6.314 GiB,缓冲只占 **34.7%**。差异有明确原因:

| | C3F(E2E 长门) | C2F(本竖条) |
|---|---|---|
| 整段臂注册的形状 | **4 个**:`[4, 4096, 8192, 16384]`(3 种 prompt 长度 + decode) | **1 个**:`[32768]` |
| 分段臂注册的形状 | **2 个**:`[4, 4096]` | **1 个**:`[4096]` |
| 缓冲收益的成因 | **形状数塌缩** + 尺寸 | **纯尺寸** |

C3F 的整段臂要为**每个不同的 prompt 长度**各留一套缓冲,所以分段的收益里
有一部分是"把 3 套变成 1 套";C2F bench 每次只跑一个长度,两臂各只有 1 套,
分段的收益**只剩尺寸**(4L 行 → 4S 行)。

**推论(对 serving 直接有用)**:C3F 的"段长整除 prompt 长度是有价值的"在
真实服务里比本竖条量到的更重要——因为真实负载有**多种 prompt 长度**,
整段路径会为每种长度各注册一套缓冲,而固定段长把它们全部塌缩成一套。
本竖条的 34.7% 是该收益的**下界**(单一长度),C3F 的 50.7% 是有 3 种长度时的值。

### 4.3 容量墙:16K 整段 **OOM**,分段 16K 只用 14.4 GiB

任务矩阵之外补了一个 16384 的点,因为"分段到底有什么用"要在容量边界上才看得清:

| L=16384 | 结果 | tok/s | peak | activation | 收尾 free |
|---|---|---:|---:|---:|---:|
| **整段** | **OOM(4 rank 全部)** | — | — | — | — |
| 8×2048 | ok | 24,308(离散 0.08%) | 15.319 | 1.827 | 7.422 |
| **16×1024** | **ok** | **24,008**(离散 0.19%) | **14.382** | **1.202** | **8.375** |

整段臂的 OOM 原文(4 个 rank 完全一致):

```
torch.OutOfMemoryError: CUDA out of memory. Tried to allocate 512.00 MiB.
GPU 0 has a total capacity of 23.52 GiB of which 401.94 MiB is free.
Including non-PyTorch memory, this process has 23.12 GiB ...
```

**§4.1 的线性模型预测了这堵墙,并且量对了**。把两项都写成段长 S 的线性式:

```
peak(S) ≈ 13.207  +  0.3126·(S/1024)   +  0.576·(S/1024) + 0.34   GiB
          └权重+KV┘   └MoE 缓冲 80 KiB/行┘   └───前向激活───┘
        = 13.207 + 0.8886·(S/1024)
```

| S | 模型 | 实测 |
|---:|---:|---:|
| 2048 | 14.98 | 14.756 |
| 4096 | 16.76 | 16.640 |
| 8192 | 20.32 | 20.409 |
| **16384** | **27.42** | **OOM(> 23.12 可用)** ✅ |

模型在 2048–8192 上误差 ≤1.5%,外推到 16384 给出 27.4 GiB —— 比可用的
23.12 GiB 高 **4.3 GiB**,与"跑到 23.12 GiB 时在一次 512 MiB 分配上失败"
的实际形态一致。反解可用容量 23.1 GiB 得
**整段 prefill 的天花板 ≈ 11.4K token**。
即:**整段路径在 8K 还有 2.277 GiB 余量,到 ~11K 撞墙,16K 必 OOM;
分段到 1024 后 16K 只用 14.382 GiB,还剩 8.375 GiB。**

这就是分段的**真正用途**,也是它唯一无可替代的地方:
它把 prefill 的峰值显存从 **O(总长)** 变成 **O(段长)**,
代价是 §1 的 1–4% 吞吐。16K 那一格的"收益"不是 1.5×,是
**从跑不了变成跑得了**。

> 顺带:16K 分段的每 token 成本(S=1024 时 166.612 µs)比 8K 分段
> (164.942)高 1.0%,继续沿着 §2.3 的上下文税曲线走,**没有出现拐点**——
> 到 16K 为止 O(s²) 项仍然不主导。

## 5. 为什么 C3F 看到 1.5–2.3×,这里看到 0.96–0.99×

C3F 的加速是**真实测到的**,本竖条不质疑那次测量。差异来自两个口径不是同一件事:

1. **C3F 整段臂在容量悬崖上**。C3F 记录整段跑 4096 时 stage 0 收尾只剩
   **0.615 GiB**;本竖条整段跑 8192 收尾还有 **2.277 GiB**,整段跑 4096 有
   **6.078 GiB**。贴着 OOM 跑的分配器行为(碎片、重试、同步)与有余量时
   完全不同,而 C3F 的分段臂把 free 拉回 7–8 GiB —— **两臂的差里混进了
   "整段臂被显存压着"这一项**。
2. **C3F 的 4 形状注册**(§4.2)让整段臂额外背了 3 套 per-shape 缓冲,
   既吃显存又加剧第 1 条。
3. **口径不同**:C3F 的墙钟是 rank 12(stage3 tp0)的**含 PP 上游等待**的
   整条流水 prefill 时延,里面有 4 个 stage 的串行与 handoff;C2F 是
   **单 stage、无 PP handoff、open-loop**。前者对"哪个 stage 先被显存拖住"
   敏感,后者不敏感。

**本竖条不推翻 C3F 的任何记录**,只给出同口径下的判定:在**没有容量压力、
单一形状注册、单 stage**的条件下,分段的内在效应是 **−1% ~ −4%**。
C3F 的 1.5–2.3× 应当理解为**"把整段臂从容量悬崖上救回来"的收益**,
而不是分段本身的算力收益 —— 这两件事在 C3F 的实验里是耦合的。

## 7. prefill/decode 同池共存评估(基于实测显存)

§6 风险表的原判是"**2 节点无 P/D 分离空间,prefill 挤占 decode**(低中)"。
本竖条给出可以直接算的两个量。

### 7.1 decode 侧的余量:实测 **0.531 GiB/卡**

8K decode 前沿(fp8 KV + fp8 indexer、mb4、**bl72** = 8,733 tok/s 那个点,
`E1F/results/workspace/out-e1if-timed-fp8-idxfp8-global-mb4-bl72-ctx8192-l1`)
的 `memory/after_build/free_bytes` = **570,294,272 B = 0.531 GiB**,
bl80 直接 OOM。实测的容量曲线:

| bl | 建图后 free (GiB) | 备注 |
|---:|---:|---|
| 56 | 2.400 | |
| 60 | 1.916 | |
| 64 | 1.410 | |
| **72** | **0.531** | **8,733 tok/s 的前沿点** |
| 80 | OOM | |

拟合斜率:**≈0.12 GiB / bl 单位**(bl56→60 是 0.121、60→64 是 0.127、
64→72 是 0.110)。

### 7.2 prefill 侧要占多少:实测 **激活 + MoE 缓冲**

同池共存时权重是**共用**的,所以 prefill 在 decode 引擎之上真正额外要的,
就是本竖条 §4 的两项:

| prefill 形态 | 激活 | MoE 缓冲 | **额外占用/卡** | 需要退让的 bl | decode 代价 |
|---|---:|---:|---:|---:|---:|
| 整段 16384 | — | — | **不成立(单跑就 OOM)** | — | — |
| 整段 8192 | 5.041 | 2.501 | **7.542 GiB** | **~63** → bl9 | 等于关掉 decode |
| 整段 4096 | 2.523 | 1.250 | **3.773 GiB** | ~31 → bl41 | 极重 |
| 2048 段 | 1.541 | 0.625 | **2.166 GiB** | ~18 → bl54 | 重 |
| **1024 段** | **0.916** | **0.313** | **1.229 GiB** | **~10** → bl62 | **≈ −8%** |

(1024 段那一行与总长无关:L=2048 / 4096 / 8192 / 16384 切成 1024 段后
额外占用分别是 1.014 / 1.086 / 1.229 / 1.515 GiB —— 只随总长缓慢爬升,
因为 KV/压缩状态随上下文长大。**取 8192 那一行作代表**。)

(bl 退让 = 额外占用 ÷ 0.12 GiB/bl;decode 代价按 E1IF 前沿
bl40=6,392 → bl72=8,733 在区间内插值,bl62 ≈ 8.0k。)

### 7.3 结论:判断**改变**,但不是"免费"

- **整段 prefill 与 decode 同池共存,结构上不成立**:要 7.54 GiB,
  而前沿只剩 0.531 GiB,差 **14×**;把 decode 退到能腾出 7.54 GiB 的
  批量,decode 已经不剩什么吞吐了。**§6 原判在整段形态下是对的。**
- **分段到 1024 后成立**:额外占用降到 **1.229 GiB**,仍然**装不进** 0.531 GiB
  的现成余量(**差 2.3×**,这点必须说清楚——分段没有把它变成免费),
  但只需 **~10 个 bl 单位**的 decode 退让(bl72 → ~bl62,decode 8,733 → ~8.0k,
  **−8%**)就能同时容纳一个 1024 段长的 prefill 工作区。
- **所以 §6 的判断应当从"无空间"改为"有空间,代价约 8% decode 吞吐,
  且必须走分段 prefill"**。分段付出的 1–4% prefill 吞吐(§1)与它换回的
  同池可行性相比,是划算的——**这正是分段的真正用途**。

### 7.4 三条必须一起记下的限制

1. **这是显存可行性,不是时间可行性**。一个 1024 段在本 stage 上要
   **~165 ms**,期间 decode 停摆。同池共存的**时延抖动**由段长决定
   (1024 段 165 ms / 2048 段 333 ms),本竖条没有测交错调度下的实际影响。
2. **decode 前沿是 fp8 KV,prefill 侧是 bf16 KV**(§0.3)。同池必须统一
   KV dtype,统一后两侧的数字都要重取。
3. **decode 引擎是 CUDA graph + 私有内存池**,prefill 是 eager。
   0.531 GiB 是**建图之后**的余量,prefill 的 1.229 GiB 只能从 KV 里出,
   不能从 graph 池里出。上表的 bl 退让已经是按这个口径算的。

## 8. 意外发现(原样记录)

1. **C3F 的 1.5–2.3× 在同口径下不但不兑现,方向相反(0.958–0.990×)。**
   这是本竖条最重要的一条。成因已归因到 C3F 整段臂的容量压力与 4 形状注册
   (§5),不是分段的内在收益——但**归因是推断,不是实测**:要坐实需在 C3F 的
   E2E 口径下跑一个"整段臂 + 只注册 1 个形状 + 有充足 free"的对照臂。
   本竖条没有跑那个对照臂。
2. **"长整段有超线性尺寸罚金"证伪,实际是规模经济**:整段每 token 成本
   1024→8192 单调**下降** 2.0%。任何基于"长 prefill 有超线性罚金"的
   设计推论都需要重看。**长整段的问题从来不是慢,是装不下**(§4.3)。
3. **每 token 成本几乎只是"单次前向行数"的函数**(同 S across L 离散 ≤1.31%)。
   这条规律很强,可以直接用来做 prefill 调度的成本模型:
   `cost(L, S) ≈ L × f(S)`,`f` = §2.2 的表。
4. **插桩开销随前向数暴涨到 +14.7%**(1024 段长臂)。`timed()` 的
   per-component `synchronize` 在短前向下代价极高。**给后续竖条的提醒:
   任何比较不同前向数的臂时,component_walls 不可直接相减**;
   本竖条因此把 headline 全部改用无插桩 p50。
5. **MoE 集合的总线效率随形状变小而下降**(4096 行 19.7–22.7 GB/s vs
   32768 行 23.4–23.8 GB/s),这是分段 MoE 桶变差的直接原因,
   本竖条新加的段形状自检把它量出来了。
6. **`prefill peak − after load` 是个比 `max_memory_allocated` 有用得多的量**。
   后者混进了 stage load 的瞬时峰值,前者才是"前向自己要多少",
   也才是同池共存能直接用的数(§7.2)。已作为
   `prefill_activation_peak_bytes` 常驻进结果 JSON。
