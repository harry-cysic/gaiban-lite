# E4F — decode 侧融合 indexer QAT 核

第三十竖条。E2F 把 B=1 decode 的 elementwise 尾巴定位为头号可攻项
（3.06 ms/replay，39.5%，2,584 个平均 1.18 µs 的核），并指出**尾巴按核数收费、
不按字节收费**——推论是"任何减少核数的融合几乎 1:1 换成时间"。

本竖条**验证这条推论**，用的是已有的、最干净的对象：C4F 为 prefill 写的融合
indexer QAT 核（Hadamard + FP4 量化/反量化，kernel 级与层级双逐位过门）。
E2F 独立测出这条链在 decode 约 **66 µs/层**（细相位表：`index_query_done`
减去 GEMV 字节），占 replay 的 4.3%——**比 TARGET §2 的先验估算"0.2–0.4%"
高一个数量级**。

**结论：推论成立。**融合核在 decode 形状下**逐位相等**，层内成对交替 A/B 实测
**−3.82%/stage（480 步逐位）**，全模型闭环 **27.740 → 28.381 tok/s（+2.31%）**。
每 ratio-4 层省 52–58 µs，与"省掉 68 次 kernel 启动"的预测一致。

这是**尾巴按核数收费**这一机制的第一个可用杠杆。全 replay 有 2,584 个
elementwise 核，本次动的只是 5 个 ratio-4 层里的这一条链；**尾巴的绝大部分
仍未拆解**。不过换算效率不是 1:1：按 §3.2 的 292.5 µs 与 E2F 的 1.18 µs/核
反推，本次实际消掉的等效核数约 250 个，**比"ratio-4 比 ratio-128 多 68 核"
×5 层 = 340 个少约 27%**——融合核自身也要花时间，且被消掉的核并非都在
关键路径上。后续估算尾巴收益应按这个折扣，不要按核数直接折算。

---

## 1. 为什么 C4F 的 90.5× 不能直接搬

C4F 的收益机制是**省带宽**：prefill 下 `[1, 8192, 64, 128]` 的链每步材料化
268 MB FP32 临时量，约 24 GB 往返对 0.268 GB 有效信息，融合把 ~90× 冗余流量
去掉。

decode 下 `index_query` 是 `[1, 1, 64, 128]` = 8192 个值 = 16 KB。**没有带宽
可省。**能省的是这条链展开成的数十次 kernel 启动（Hadamard 的 7 级蝶形 +
FP4 量化的嵌套 `where/clamp/exp2` 链），每次都坐在 4090 的最小 kernel 时长
（~1 µs）上。**同一个核，完全不同的收益机制**——所以 90.5× 不可
折算，必须重测。

## 2. 实现

`Ratio4TorchAttention` 原本硬写 `fp4_quant_dequant(hadamard_transform(...))`
（两个 decode 调用点）。加 `indexer_qat_mode`：

- 默认 `"ref"`，**不改变任何既有行为**；
- env `DSV4_INDEXER_QAT_DECODE=fused` 或构造参数可切换；
- 只在 `[..., 128]` 宽度且 `group_size=32` 时走融合核，否则回落 eager
  （核本身只支持这一种几何）。

prefill 侧的 `Ratio4FullPositionAttention` 早有同名开关，但带
`and seqlen > 1` 的守卫，即**显式排除 decode**。本竖条解除的正是这一条，
并给出解除的依据。

## 3. 结果

### 3.1 微基准：判活/判死（不折算收益）

TARGET §9.4：微基准只用于判活/判死。口径是两条 CUDA graph 背靠背交替重放
（§9.1），外加一条只含一个平凡 kernel 的 **null 臂**来量测 `replay + event +
synchronize` 的测量地板——两条真实臂都付这个地板，所以**差值可信、比值不可信**。

titan065 单卡，shape `[1, 1, 64, 128]`，5 轮 × 200 次：

| 臂 | p50 | 减去地板 |
|---|---:|---:|
| eager 链 | 68.61 µs | **55.26 µs** |
| 融合核 | 13.36 µs | 低于地板分辨率（<1 µs） |
| null（测量地板） | 13.34 µs | — |

- **省 55.2 µs/次调用**；轮间离散 eager 0.00%、fused 0.84%、null 0.72%（均 <1%，
  满足 §9.5）。
- **逐位相等**：`max_abs_diff = 0.0`，另过 `bitwise_selfcheck`。
- 比值不报：融合臂落在 null 臂噪声内，除出来的数没有意义。

eager 的 68.61 µs 与 E2F 用完全独立的两条路子得到的估算吻合
（nsys 数核：ratio-4 比 ratio-128 多 68 核；细相位表：`index_query_done`
减去 GEMV 字节后余 ~66 µs）。**三条独立读数互相印证。**

### 3.2 层内 A/B：两条常驻 lane 交替重放

微基准不折算收益，所以层内必须实测。为此给 E2F 探针加了 `--mode ab`：

- **两条常驻 lane**（同一份权重、各自的 state/plan/图族），每步背靠背重放、
  **逐步交换先后顺序**——TARGET §9.1 规定的口径，因为本次要测的效应量
  （~3.6%）正落在"串行 A/B 不可信"的量级里；
- 每步顺带 `torch.equal` 对拍两条 lane 的 stage 输出，**这就是层内数值门**：
  微基准逐位不等于层内逐位，要在真实权重、真实 state 上再证一次。

⚠️ 踩到的坑：**MoE 对象在 lane 之间是共享的**（`physical_stage` 只给每条 lane
自己的 state/attention），所以两条 graph lane 不能都用 slots 1-3，变体 lane 需要
独立 slot 区间（本探针用 5-7，`slots_per_shape=8`），**且变体 lane 的 warmup
也必须用它自己的 slot**，否则会把基线 lane 的 slot 弄脏、让基线的 capture 以
`slot is not clean` 失败。

**对照臂（`--ab-variant none`）**：两条 lane 完全相同、不施加任何处理，其余口径
不变。这是必需的控制——两条 lane 并不完全对称（lane B 后建，张量在分配器里更
靠后；MoE slot 区间也不同），若这份不对称本身就能移动读数，那么用这套口径测的
**每一个**处理效应都会带同样的偏差。

| 臂 | base lane | variant lane | 差 |
|---|---:|---:|---:|
| `none`（对照） | 7.3605 ms | 7.3595 ms | **−0.01%** |
| `qat_fused`（处理） | 7.6489 | 7.3564 | **−3.82%** |

**相差 380 倍。**lane 不对称不是解释，−3.82% 全部来自处理本身。
（对照臂两条 lane 都读 7.36，与处理臂的 variant lane 7.3564 一致——此时默认
已是 fused，两条都在跑融合核，这本身又是一次默认生效的旁证。）

titan065，stage 0（11 层，其中 5 个 ratio-4），B=1，max_seq 3328，3 轮 ×160 步：

| 项 | 值 |
|---|---:|
| 基线 lane p50 | 7.6489 ms |
| 变体 lane p50 | **7.3564 ms** |
| 差 | **−0.2925 ms（−3.82%）** |
| 轮间离散 | base 0.209% / variant 0.233%（均 <1%，§9.5） |
| **层内逐位** | **480/480 步，max_abs_diff 0.0** |
| 变体作用层 | L2, 4, 6, 8, 10（= 全部 5 个 ratio-4 层）✓ |
| SM 时钟 | 三轮均 2760 MHz（56–58 °C） |

**每 ratio-4 层省 58.5 µs。**

值得记一笔：微基准推得的预期是 5 × 55.2 = 276 µs，层内实测 292.5 µs——
**层内不但没缩水，还略高于微基准**。这与 §9.4 列举的方向相反（C4F 的 cast 提取
微基准预期 0.65 ms/层、层内只兑现 0.166；A5F 的 2.92× 集成态只回收 4.7 ms）。
**不对成因下结论**（候选：层内还顺带省掉了这条链的分配器/临时量压力），
只记录这条反例——它说明 §9.4 的"微基准只用于判活/判死"是双向的，
不能默认层内一定更小。

### 3.3 全模型闭环（16 卡 PP4，E1F 冻结脚本）

层内 A/B 只覆盖 stage 0 的 5 个 ratio-4 层；全模型有 21 个。用 E1F 的冻结脚本
（`run_e1f_dual.sh 1 bitwise 3 300 2048`）加 `DSV4_INDEXER_QAT_DECODE=fused`：

| 项 | 基线（本 session 复跑） | fused | 差 |
|---|---:|---:|---:|
| **单路 decode** | 27.740 tok/s | **28.381 tok/s** | **+2.31%** |
| stage 0 replay（5 个 r4） | 8.2247 ms | 8.0028 | −0.2219 |
| stage 1 replay（5 个 r4） | 8.2682 | 8.0259 | −0.2423 |
| stage 2 replay（6 个 r4） | 8.4959 | 8.1449 | −0.3510 |
| stage 3 replay（5 个 r4） | 7.5383 | 7.2569 | −0.2814 |
| 四级合计 | 32.5272 | 31.4305 | **−1.0967** |
| step wall | 36.085 | 35.226 | −0.859 |
| settle 逐位（graph vs eager） | — | 132/132 | — |

**每 ratio-4 层 −52.2 µs**（1096.6 µs ÷ 21），是隔离 A/B 的 −58.5 µs 的 89%
——集成态略低于隔离态，方向正常。

⚠️ **口径声明**：这两次是**两个独立的 16 卡作业**，属串行 A/B。本 session 实测
16 卡作业的**跑间离散约 1.0%**（见 §3.4），而此处效应量 2.31% 只有它的 ~2.3 倍。
**因果结论不建立在这一对数上**，而建立在 §3.2 的成对交替 A/B（离散 0.21%、
四个 stage 的每层边际一致）；本节只是确认它在全模型闭环里确实兑现、量级一致。

### 3.4 一次意外得到的 16 卡跑间离散，以及一个静默失效的教训

第一次跑"fused 闭环"时读到 27.459 tok/s，**比基线 27.740 还慢 1.0%**，
per-stage replay 与基线几乎逐项相同。与 §3.2 的 −3.82% 直接矛盾。

**成因不是杠杆无效，是杠杆根本没打开。**`run_*.sh` 的 `ENV_BASE` 是单引号
字符串，往里插的 `${E1F_EXTRA_ENV:-:}` 不在本地展开，作为字面量传到远端；
远端没有该变量，展开成 `:`（空操作）。**症状与"该杠杆无效"完全同形。**

两项修正：

1. 插入点改为让本地展开：`ENV_BASE='...'"; ${VAR:-:}"`；
2. **给 `e1f_full_decode_bench.py` 加处理组见证** `attention_modes`——记录每层
   实际解析出的 `indexer_qat_mode`。重跑后 artifact 里可见
   `{'2': {'indexer_qat_mode': 'fused'}, '4': {...}, '10': {...}}`
   而 L0/L1/L3（window / ratio-128）为空，与预期一致。

**副产品**：那次失败的运行其实是一次合格的**第二基线**（27.459 vs 27.740），
于是有了一个此前没有的数：**16 卡闭环作业的跑间离散 ~1.0%**——比隔离 4 卡平台的
0.2% 大 5 倍。这解释了为什么 §3.3 只能作确认、不能作因果。

这条教训已写入 TARGET §9.10 与 HANDOFF §3.0.7：**A/B 的结果 JSON 必须记录
实际解析出的处理组配置，否则"没生效"会被当成"没效果"写进结论。**

### 3.5 D0L 长门

本改动是**逐位**的，不是语义变更——三级证据：kernel 自检、decode 形状微基准
（max_abs_diff 0.0）、层内 480 步逐位。但"逐位所以门必过"是推理不是实测，
所以仍跑一次冻结长门（`run_e0l2e_long_arm.sh qatfused eager decode` +
`DSV4_INDEXER_QAT_DECODE=fused`）：

| 项 | 冻结基线 | fused |
|---|---:|---:|
| 命中 | 494/512 | **494/512** |
| match_rate | 0.964844 | **0.964844** |
| max `top2_gap` | 0.959503173828125 | **0.959503173828125** |
| median / min gap | 0.345806… / 0.000545… | **逐位相同** |
| 八条 prompt 的首个失配 | step/pred/golden/logit | **逐位相同** |
| accepted | True | **True** |

**不是"过门"，是"输出逐位一致"**——正是一个逐位改动应当产生的结果。
两条放行判据（分数不降、gap 不越包络）都不是"勉强通过"，而是同一个数。

### 3.6 已设为默认

证据链完整（kernel 自检 → decode 形状 max_abs_diff 0.0 → 层内 480 步 →
D0L 长门逐位一致）且单路 +2.31%，故**默认改为 `fused`**；
`DSV4_INDEXER_QAT_DECODE=ref` 可回退到未融合链。

翻默认之后不带任何 env 再跑一次 E1F 验证它确实生效：

| 项 | 值 |
|---|---:|
| 见证 `attention_modes` L2（ratio-4） | `{'indexer_qat_mode': 'fused'}` |
| 见证 L3（ratio-128，不应有该字段） | `{}` |
| tok/s p50（无 env） | **28.402** |
| accepted | True |

与带 env 的 28.381 一致（差 0.07%），确认默认路径生效而非靠 env 兜住。

⚠️ **对既有数字的影响**：E4F 之前的所有 decode 实测（含 E1F 的 8,733 @8K 等）
都是在**未融合**下测的。本改动是逐位的，所以**质量口径完全不受影响**；
但**速度口径**在 E4F 之后会略高，跨 E4F 比较 decode 速度时须注明。
大 B 下收益应当远小于 B=1（这条链是每步固定成本，大 B 摊薄），**未测**。
